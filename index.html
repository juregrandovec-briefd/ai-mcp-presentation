What Are AI Coding Agents?

Autonomous systems that execute complete development workflows

Key Characteristics:

  • Execute multi-step workflows
  • Make autonomous decisions
  • Use tools and APIs
  • Iterate until task complete
  • Ask for approval at each step

4. Create tests [approve]
5. Run tests [approve]

Core Prompting Principles (2025)

1. Be Explicit with Instructions
‚Ä¢ Go beyond basics - request fully-featured implementations
‚Ä¢ Describe exactly what you want in output
‚Ä¢ Example: "Create comprehensive OAuth implementation with error handling, logging, and edge case coverage"
2. Add Context and Motivation
‚Ä¢ Explain WHY certain behaviors matter
‚Ä¢ Help AI understand underlying goals
‚Ä¢ Example: "We need caching because API has rate limits and user experience degrades above 200ms"
3. Precision Over Cleverness
‚Ä¢ Clear structure beats clever wording
‚Ä¢ Most failures come from ambiguity, not model limitations
‚Ä¢ Use structured formats (XML tags, JSON) for complex requests
TIMING: 1.5 minutes
  • These are the three core principles from Claude's official 2025 documentation
  • Principle 1: Be explicit with instructions - Go beyond basics, request fully-featured implementations. Don't just say "add login", say "create comprehensive OAuth with error handling, logging, and edge case coverage"
  • Principle 2: Add context and motivation - Explain WHY certain behaviors matter. Example: "We need caching because API has rate limits and user experience degrades above 200ms"
  • Principle 3: Precision over cleverness - Clear structure beats clever wording. Most failures come from ambiguity, not model limitations
  • Use structured formats like XML tags or JSON for complex requests
  • Remember: Most prompt failures are from ambiguity, not AI limitations

Complete Prompt Structure

7-Part Template:

1. Context: What's the situation?

2. Task: What needs to be done?

3. Constraints: Any limitations?

4. Output format: How should it respond?

5. Expected Behavior: Describe edge cases

6. Success Criteria: Define "done"

7. Reasoning Request: Ask agent to explain approach first

Real-World Example:
Context: Laravel API with 100K+ daily users
Task: Add rate limiting to prevent abuse
Constraints: Redis available, must not break existing auth
Output: Implementation + migration guide + monitoring dashboard
Expected: Handle burst traffic, graceful degradation
Success: 429 responses after limit, <5ms overhead
Reasoning: "Explain your approach before coding"
TIMING: 1.5 minutes
  • The complete prompt structure has 7 parts - walk through each one from the slide
  • Parts 1-4 are basics: Context (what's the situation?), Task (what needs doing?), Constraints (limitations?), Output format (how should it respond?)
  • Parts 5-7 make all the difference: Expected Behavior covers edge cases, Success Criteria defines what "done" means, Reasoning Request asks agent to explain approach first
  • Show the real-world example: Laravel API with 100K+ daily users, adding rate limiting
  • Point out the specificity: Redis, 429 responses, less than 5ms overhead, monitoring dashboard, graceful degradation
  • Emphasize: We're not just saying "add rate limiting" - we're providing complete context, constraints, and success criteria

Good vs Bad Prompts & Common Pitfalls

‚ùå Bad Prompts

"Fix the bug" - Too vague, no context
"Make it better" - Subjective, no direction
"Add 6+ features at once" - Too many tasks

‚ùå Common Pitfalls

Ignoring agent questions - Answer clarifications
Not reviewing output - Always review & test
Expecting perfection - Iterate & refine

‚úÖ Good Prompts

"Fix UserController line 45: email validation should allow + characters"
Specific location + requirement
"Refactor calculateTotal() to O(n), maintain same API"
Clear goal + constraints
"Implement OAuth2: 1) Setup, 2) Backend, 3) Frontend, 4) Tests"
Complex task broken down
Complete Example:
‚ùå BAD: "Add error handling"
‚úÖ GOOD: "Add error handling to UserService: try-catch on DB calls, log to Sentry with context, return standardized JSON errors, retry transient failures (3x, exponential backoff), write tests, update API docs"
TIMING: 2 minutes
  • This slide combines bad prompts and common pitfalls
  • Bad prompts - Left side top: "Fix the bug" is too vague, "Make it better" is subjective, "Add 6+ features at once" is too many tasks
  • Common pitfalls - Left side bottom: Ignoring agent questions (always answer clarifications!), Not reviewing output (always review and test!), Expecting perfection (iterate and refine!)
  • Good prompts - Right side: Specific location + requirement ("Fix UserController line 45"), Clear goal + constraints ("Refactor to O(n)"), Complex task broken down into steps (OAuth: 1-2-3-4)
  • Show the bottom complete example: "Add error handling" BAD versus detailed specification with Sentry logging, retry logic with exponential backoff, standardized JSON errors, tests, and docs

TIMING: 1.5 minutes

  • Output Verification: Request sources to minimize hallucinations - "Provide file paths and line numbers", "Cite documentation sources", "Show relevant code context"
